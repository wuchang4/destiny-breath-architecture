# 5个顶级 AI Agent 项目分析 — 可借鉴的设计模式

> 基于 2026 年 5 个 GitHub 高星项目的架构分析，提取可直接融合进天命·生息架构的设计模式。

## 一、DeerFlow 2.0（字节跳动）— 65K Stars

**定位：** Deep Research 框架，Multi-Agent 调研员

### 核心架构

```
主代理（Coordinator）— 调度器
    ├── Planner（任务规划器）
    ├── Researcher（研究员）× N
    ├── Coder（代码执行代理）
    └── Reporter（报告生成器）
```

### 可借鉴的设计

| 天命现状 | DeerFlow 做法 | 差距 |
|---------|-------------|------|
| 天行军有斥候/谋士/文官固定角色 | **子代理池动态拉起**：按任务复杂度动态伸缩代理数量 | 天行军角色固定，不能自动扩展 |
| 三省图线性流水线 | **四阶段迭代循环**：Plan→Research→Revise→Deliver | 天行军没有迭代机制，一次性执行 |
| 一个搜索后端 | **搜索策略抽象层**：Tavily/DDG/Brave/SearxNG 热插拔 | 集成多个搜索源做交叉验证 |
| 无沙箱 | **AIO 沙箱**：Docker 内 Shell+File+Browser+MCP+VSCode | 安全执行环境缺失 |

**最值得借鉴：**
1. **子代理池动态伸缩** — 简单任务1个代理，复杂任务拉起几十个
2. **四阶段迭代** — 不要一次做对，Plan→Execute→Revise→Deliver 循环推进
3. **搜索策略抽象层** — 多个搜索源统一接口，交叉验证

## 二、Prompt-Optimizer — 28K Stars

**定位：** AI 提示词优化工具，大白话指令转专业指令

### 核心能力

- 输入自然语言 → 输出结构化优化提示词
- 支持多轮迭代改进
- 实时对比优化前后效果
- 纯前端实现，数据本地处理

### 可借鉴的设计

| 天子令现状 | Prompt-Optimizer 做法 | 差距 |
|-----------|---------------------|------|
| 无结构化提示词生成 | **一键优化**：输入"写小红书爆款"→ 输出含标签/钩子/结构化的完整Prompt | 需要手动写复杂指令 |
| 无对比测试 | **A/B测试**：实时对比优化前后的效果差异 | 没有量化"优化是否有效"的手段 |

**最值得借鉴：**
1. **自然语言→结构化Prompt的转换能力** — 但不是核心架构需求，是工具级功能

## 三、Agents（agency-agents）— 35K Stars

**定位：** AI 员工角色库，80+ 专业角色

### 核心设计

```
┌─────────────────────────┐
│      agent 定义文件      │
│ 每个角色 = 一个 Markdown │
│ role: 前端工程师          │
│ skills: React/Vue/CSS   │
│ tools: code_edit/lint   │
│ system_prompt: ...       │
└─────────────────────────┘
```

### 可借鉴的设计

| 天子令现状 | Agents 做法 | 差距 |
|-----------|------------|------|
| 天行军有固定角色 | **80+角色即插即用** | 需要大量预设角色定义 |
| 角色定义在代码中 | **角色 = Markdown 文件** | 解耦了角色定义和执行代码 |

**最值得借鉴：**
1. **角色定义的 Markdown 化** — 每个角色一个 Markdown 文件，解耦定义和执行
2. **角色即插即用** — 按需加载角色，不用时零成本

## 四、Ruflo（原 Claude-Flow）— 45.7K Stars

**定位：** 多 Agent 编排平台，项目级任务拆解与分配

### 核心架构

```
Ruflo Orchestrator
    ├── 任务拆解引擎（Complex → Subtasks）
    ├── 角色分配器（谁最适合做什么）
    ├── 验收引擎（自动验证结果）
    ├── 自学习记忆系统（跨会话）
    └── 集群协调（多机器/多团队）
```

### 可借鉴的设计

| 天子令现状 | Ruflo 做法 | 差距 |
|-----------|-----------|------|
| 三省图线性执行 | **任务树拆解**：一个复杂任务拆成 10+ 个子任务并行 | 天行军不能处理超复杂任务 |
| 无自动验收 | **自动验收引擎**：子任务完成后自动检查质量 | 缺少质量门禁 |
| 三省图手动执行 | **集群协调**：多个 Agent 跨机器协作 | 单机单实例限制 |

**最值得借鉴：**
1. **自动验收引擎** — 子任务完成后自动验收，不合格回退重做
2. **任务树并行执行** — 不是线性流水线，是树形拆解+并行执行

## 五、Letta（前身 MemGPT）— 22.5K Stars

**定位：** 有状态 Agent 平台，让 Agent 拥有持久化记忆

### 核心架构

```
LLM 视角：虚拟无限上下文（LLM 以为自己能看到全部）
    ↕
Letta 调度器（上下文调度器）
    ↕
当前上下文（LLM 直接可见）
    ├── Core Memory（human/persona/custom blocks）
    └── 对话历史
    ↕
Archival Memory（向量存储，LLM 不可直接访问）
    ├── 长期记忆
    ├── 历史对话（压缩存储）
    └── 知识文档
```

### 可借鉴的设计

| 天子令现状（五层记忆） | Letta 做法 | 差距 |
|---------------------|-----------|------|
| 五层手动记忆 | **Memory Blocks**：个人/人设/自定义块，结构化 | 天命的五层偏重"存储位置"，Letta偏重"结构" |
| 无重要性评分 | **重要性评分+Compaction**：LLM自己判断记忆重要性，自动压缩 | 天命的蒸馏是人手工的 |
| 无跨会话自动持久化 | **状态持久化Agent**：Agent是有状态的，跨会话保持 | 天命每次会话是新的 |
| 无虚拟上下文管理 | **分页调度**：Core→检索→换入→换出，像OS管理内存 | 上下文满了就截断，没有智能管理 |

**最值得借鉴：**
1. **Memory Blocks 结构化** — 把记忆按标签（human/persona/custom）分块，比现在的大文本文件更好
2. **重要性评分 + 自动 Compaction** — LLM 自己决定什么重要、什么可以忘记
3. **分页上下文管理** — Core Memory 常驻 + Archival 检索 + 自动换入换出
4. **跨会话状态持久化** — Agent 是有状态的，不是每次对话从头开始

## 六、天命架构的融合路线图

### P0（当前正在做）：仪表盘诚实 + 电路断路器
✅ 已完成

### P1（优先融合）：记忆系统升级 — 借鉴 Letta

```
当前五层记忆：
  底层 SOUL.md → 底层 leta（架构规则，不可篡改）
  深层 REVIEW.md → 深层（复盘记录）
  中层 MEMORY.md → 中层（用户偏好/索引）
  向量层 vector_memory.py → 向量层（语义搜索）
  表层 daily log → 表层（当前会话）
  
借鉴 Letta 改为 Block 结构：
  Core Blocks（常驻上下文）：
    ├── persona block  → SOUL.md（我的人设）
    ├── human block    → USER.md（你的信息）
    └── user_prefs block → MEMORY.md（偏好/习惯）
  
  Archival Memory（向量存储，按需检索）：
    ├── 历史记忆（REVIEW.md + 旧日志）
    ├── 技能记忆（SKILL.md 索引）
    └── 决策记录（ROUTING_LOG.md）

  Compaction（自动压缩）：
    当 Core Memory 太大时 → LLM 自评重要性 → 低频低价值信息降入 Archival
```

### P2（中期）：执行引擎迭代 — 借鉴 DeerFlow + Ruflo

```
当前 destiny_engine.py（三省图线性执行）：
  中书省 → 门下省 → 尚书省 → 执行 → AAR

借鉴后的执行模式：
  主代理解析任务
    ├── 子任务1（独立 Agent 执行）
    ├── 子任务2（独立 Agent 执行）  ← 并行
    ├── 子任务3（独立 Agent 执行）
    └── 自动验收引擎检查每个子任务
        不合格 → 回退重做
        合格   → 汇总到最终输出
```

### P3（长期）：角色 Markdown 化 + 搜索抽象层

- 天行军的斥候/谋士/文官 改为 Markdown 文件定义
- 搜索后端接入 Tavily/Brave/SearxNG 等，统一接口
