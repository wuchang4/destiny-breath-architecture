## 二、优化方案（按优先级排序）

### P0：错误处理与恢复（借鉴生产最佳实践）

**现状：** 三省图有 interrupt/resume，但没有错误重试、电路断路器、优雅降级。

**目标：** 让架构在 API 挂了、工具超时、模型不可用时不会直接崩溃。

#### 2.1 错误分类系统

```
错误类型        → 处理策略         → 天命实现
───────────────────────────────────────────────
临时错误         → 指数退避重试     → 新增 RetryPolicy 类
(网络超时/429/503)

可恢复错误       → 结构化回退       → 新增 FallbackChain 类
(格式错误/参数无效/上下文溢出)

致命错误         → 优雅降级+通知    → 已有 interrupt/resume
(认证失败/配额耗尽/安全违规)        + 新增 CircuitBreaker
```

#### 2.2 电路断路器（新增组件）

借鉴 Claw Code 的 provider 检测 + 生产最佳实践的 Circuit Breaker 模式：

```python
# 对每个外部 API 维护一个断路器
# 连续失败 3 次 → OPEN（拒绝请求，直接走降级）
# 60 秒后 → HALF_OPEN（试探一次）
# 试探成功 → CLOSED（恢复正常）
```

**集成点：** model_router.py 的 `check_availability()` 方法中加入断路器状态。

#### 2.3 优雅降级链（新增组件）

```
主模型不可用 → 降级到本地模型 → 降级到缓存响应 → 降级到模板回复 → 诚实告知用户
工具不可用   → 降级到同功能工具 → 降级到手动操作指引 → 诚实告知用户
网络不可用   → 降级到离线模式 → 降级到本地知识库 → 诚实告知用户
```

**集成点：** destiny_engine.py 的执行节点中加入降级逻辑。

### P2：记忆系统深度整合

**现状：** 五层记忆系统有设计，但表层→中层的蒸馏是手动的，中层→深层的固化几乎没发生。

**优化方案：**

```
表层（daily log）→ 自动蒸馏规则（已有，未自动触发）
  触发条件：
    - 同一话题跨3次会话被引用 → 自动提取写入 MEMORY.md
    - 用户明确说"记住" → 立即写入
    - 会话结束时 → 检查是否有值得保留的信息

向量层（semantic search）→ 需要实际使用
  当前只在 Protocol 4 早朝中提及，未真正使用
  需要在每次回答前自动执行语义搜索

中层（MEMORY.md）→ 手动 → 半自动
  需要 cron 触发日志蒸馏

深层（REVIEW.md/ROUTING_LOG.md）→ 需要实际写入
  AAR 的复盘记录从未写入过 REVIEW.md

底层（SOUL.md）→ 需要更严格的修改门槛
  今天的教训：一次做对不够，需要多次验证
```

### P3：可观测性（借鉴 OpenTelemetry）

**现状：** 有 execution_tracer.py（span/trace），但没有持久化、没有聚合、没有告警。

**优化方案：**

```
当前：span 记录到内存 → 导出到单个 JSON 文件
目标：span 记录 → SQLite 持久化 → 按任务聚合 → 趋势分析 → 异常告警

具体实现：
1. 新增 trace_store.py：span 写入 SQLite 而非单个 JSON
2. 新增 trace_analyzer.py：按任务类型聚合，计算 P50/P95 延迟
3. 新增 trace_dashboard.py：Web UI 展示执行追踪
4. 在 heartbeat.py 中加入追踪分析：检测异常延迟并告警
```

---

### P4：三省图与对话流真正绑定

**现状：** destiny_engine.py 是独立 CLI 工具，没有被任何 Skill 或 Protocol 自动调用。

**优化方案：**

```
方案 A：创建 "destiny-execution" Skill
  - 当用户发出需要执行的任务时，自动调用 destiny_engine.py
  - Skill 的 SKILL.md 中写明：每个非简单任务必须走引擎
  - 触发条件：涉及工具调用的任务

方案 B：修改 Protocol 0（上朝公示）
  - 在 Protocol 0 的执行规则中加入：接令后先调用 destiny_engine.py
  - 通过 model_router 选模型
  - 通过 tool_safety_chain 做安全验证
  - 通过 execution_tracer 记录追踪
  - 执行完毕后自动写检查点

推荐：方案 B（更彻底，但改动更大）
```

---

### P5：自进化引擎真正运行

**现状：** 4 个引擎有代码，但从未真正用 metric_score 驱动过优化。

**需要实现的闭环：**

```
每次任务结束
    ↓
记录 metric_score（自评 or 用户反馈）
    ↓
对比基线（同类型任务的历史分数）
    ↓
如果 delta < 0 且连续 ≥3 次
    ↓
触发梯度反向传播（textgrad 逻辑）
    ↓
更新 Logic Anchors 或 MEMORY.md
    ↓
验证：下次同类任务是否改善
```

**当前缺失：** "记录 metric_score" 和 "对比基线" 这两步从未执行过。
